# Antibody Drug Tracker — Project Instructions

This file records all design decisions, API details, known bugs fixed, and limitations
discovered during development. Use it as the primary reference for future updates or debugging.

---

## Project Overview

A publicly shareable, auto-updating website cataloging antibody-based therapeutics
(monoclonal antibodies, ADCs, bispecifics, nanobodies, antibody fragments, etc.)
across three categories. Hosted free on GitHub Pages, auto-refreshed every 2 weeks
via GitHub Actions.

**Live URL:** https://alex810918.github.io/antibody-tracker/
**Repository:** https://github.com/alex810918/antibody-tracker

---

## Architecture

```
antibody-tracker/
├── .github/workflows/update-data.yml   ← GitHub Actions: runs 1st & 15th of each month
├── data/
│   ├── category1.json                  ← Marketed drugs (generated by script)
│   ├── category2.json                  ← Active clinical trials (generated by script)
│   ├── category3.json                  ← Failed programs 2021–2026 (generated by script)
│   └── last_updated.json               ← Timestamp of last successful run
├── scripts/
│   ├── fetch_category1.py              ← FDA BLA applications via openFDA
│   ├── fetch_category2.py              ← Active trials from ClinicalTrials.gov
│   ├── fetch_category3.py              ← Terminated/withdrawn from ClinicalTrials.gov
│   ├── fetch_revenue.py                ← 2025 revenue enrichment via SEC EDGAR
│   └── update_all.py                   ← Orchestrator: runs all scripts in order
├── index.html                          ← 3-tab website (no framework)
├── style.css
├── app.js
└── INSTRUCTIONS.md                     ← This file
```

**Tech stack:**
- Python 3.12+ with `requests` library (only external dependency)
- GitHub Pages for hosting (free, public URL)
- GitHub Actions for scheduling (free)
- Plain HTML/CSS/JavaScript — no build step, no framework

**To add a new data source in the future:**
1. Create `scripts/fetch_categoryN.py` with a `main()` function
2. Add the script name to the `SCRIPTS` list in `update_all.py`
3. Add the JSON field to the `TABS` config in `app.js`
4. Add an HTML tab button + panel + filter selects in `index.html`

## Data Sources

### Category 1 — Marketed Antibody Drugs

**Source:** openFDA Drugs@FDA endpoint
**API URL:** `https://api.fda.gov/drug/drugsfda.json`
**Query:** `search=application_number:BLA*&limit=100&skip=N`
**Free:** Yes, no API key required
**Rate limit:** ~240 requests/minute; script uses 0.3s sleep between pages

**How it works:**
- Fetches all 466 BLA (Biologics License Application) records from openFDA
- Filters client-side using antibody keyword matching on brand name and INN (active ingredient)
- Keywords: `mab`, `monoclonal`, `antibody`, `bispecific`, `adc`, `antibody-drug conjugate`, `fab`, `scfv`, `nanobody`, `immunoglobulin`
- INN stems: `mab`, `ximab`, `zumab`, `umab`, `imab`, `tumab`, `lumab`, `mumab`, `lizumab`, `tuzumab`

**Fields available from openFDA:**
- Brand name, INN/active ingredient, sponsor name, application number
- Approval date (derived from submissions where `submission_type=ORIG` and `submission_status=AP`)
- Marketing status (used as a proxy for patent status — limited accuracy)

**Fields NOT available from openFDA:**
- Indication (not reliably structured in this endpoint)
- Patent expiry date (not in this endpoint)
- Revenue (see fetch_revenue.py)

**Known bug fixed:** openFDA returns HTTP 404 (not an error — means 0 results) when no records match.
The `safe_get()` helper returns `{}` for 404 rather than raising an exception.

**Known bug fixed:** Wildcard queries must NOT use quotes.
`search=application_number:"BLA*"` → broken (returns 0 results)
`search=application_number:BLA*` → correct (returns 466 records)

### Category 2 — Active Clinical Trials

**Source:** ClinicalTrials.gov API v2
**API URL:** `https://clinicaltrials.gov/api/v2/studies`
**Free:** Yes, no API key required
**Rate limit:** Polite use; script uses 0.3s sleep between pages

**Query parameters used:**
```
query.intr = "monoclonal antibody OR bispecific antibody OR antibody-drug conjugate OR ADC OR mAb OR nanobody OR immunoglobulin therapy"
filter.overallStatus = RECRUITING|ACTIVE_NOT_RECRUITING|NOT_YET_RECRUITING|ENROLLING_BY_INVITATION
filter.advanced = AREA[InterventionType]BIOLOGICAL
fields = NCTId, BriefTitle, OfficialTitle, OverallStatus, Phase, LeadSponsorName,
         Condition, InterventionName, InterventionType, StartDate, PrimaryCompletionDate
pageSize = 100
format = json
```

**Pagination:** Uses `nextPageToken` from each response.

**Fields extracted:**
- NCT ID (linked to ClinicalTrials.gov study page)
- Drug name (from BIOLOGICAL-type interventions; falls back to study title)
- Sponsor (LeadSponsor)
- Phase (normalized from PHASE1/PHASE2/PHASE3 to "Phase 1/2/3")
- Status
- Indication (top 3 conditions joined with "; ")

**Current record count:** ~1,131 active studies

### Category 3 — Failed Programs (2021–2026)

**Source:** ClinicalTrials.gov API v2 (same endpoint as Category 2)
**Same API URL and auth as Category 2**

**Query parameters used:**
```
query.intr = (same antibody keywords as Category 2)
filter.overallStatus = TERMINATED|WITHDRAWN
fields = NCTId, BriefTitle, OfficialTitle, OverallStatus, Phase,
         LeadSponsorName, Condition, InterventionName, InterventionType,
         WhyStopped, StartDate, CompletionDate, LastUpdatePostDate
pageSize = 100
format = json
```

**Date filtering (2021–2026):** Done client-side by reading `LastUpdatePostDate` from each study. Records outside 2021–2026 are excluded during parsing.

**Known bug fixed:** `filter.lastUpdatePostDate` is NOT a valid ClinicalTrials.gov v2 API parameter — it causes a 400 Bad Request error. Removed; date filtering is now done client-side.

**Known bug fixed:** `filter.advanced=AREA[InterventionType]BIOLOGICAL` combined with `filter.lastUpdatePostDate` caused 400 errors. The `filter.advanced` parameter works fine alone (kept for Category 2 but removed from Category 3 to avoid the conflict).

**Failure type classification logic (in `_classify_reason()`):**
- `WhyStopped` field contains free-text from the sponsor
- Keywords in `WhyStopped` are matched to classify into:
  - `"FDA rejection (CRL)"` — if "fda", "crl", "complete response", "refuse to file", "regulatory" appear
  - `"Sponsor withdrawal"` with sub-reason:
    - Safety: "safety", "adverse", "toxicity", "death", "serious"
    - Efficacy: "efficacy", "futility", "interim analysis", "did not meet", "lack of"
    - Business: "business", "financial", "funding", "strategic", "portfolio"
    - Enrollment: "enroll", "recruitment", "accrual", "feasibility"
  - `"Unknown"` → displayed as "Reason not publicly disclosed"

**openFDA CRL supplement:** Also queries openFDA for formal FDA rejections (Complete Response Letters) on BLA applications.
Query: `search=application_number:BLA* AND submissions.action_type:CRL`
**Status:** Currently returns 0 results (openFDA returns 404 / no matches). This is a known limitation — the nested query against `submissions.action_type` may not be indexed the same way.

**Current record count:** ~1,409 terminated/withdrawn studies

### Revenue Enrichment (fetch_revenue.py)

**Source:** SEC EDGAR
**API URLs:**
- Company search: `https://www.sec.gov/files/company_tickers.json` (ticker → CIK mapping)
- XBRL facts: `https://data.sec.gov/api/xbrl/companyfacts/CIK{cik:010d}.json`
- Full-text search: `https://efts.sec.gov/LATEST/search-index`
**Free:** Yes. EDGAR requires a `User-Agent` header with name + email (enforced).
**Header used:** `User-Agent: antibody-tracker research@example.com`

**How it works:**
1. Reads `category1.json` (must run after `fetch_category1.py`)
2. For each drug, looks up parent company's SEC CIK number
3. First tries XBRL structured facts for 2025 10-K revenue
4. Falls back to full-text search of SEC filings for dollar amounts near the drug name
5. Writes `revenue_2025` field back into `category1.json`

**Company → ticker override map** (hardcoded in script, update as needed):
- Roche/Genentech → RHHBY, AbbVie → ABBV, J&J/Janssen → JNJ, Merck → MRK,
  BMS → BMY, Pfizer → PFE, AstraZeneca → AZN, Novartis → NVS, Lilly → LLY,
  Amgen → AMGN, Regeneron → REGN, Sanofi → SNY, Biogen → BIIB, Gilead → GILD

**Known limitation:** Product-level revenue is only disclosed when material under SEC rules. Most companies report therapeutic area totals, not individual drug revenue. Current hit rate: 0/218 (all show "Not available from public filings").
This is expected behavior — the module is a placeholder for future improvement.

**Future improvement options:**
- Parse 10-K MD&A sections with a more sophisticated text extractor
- Use a paid data source (e.g., Evaluate Pharma, IQVIA) for reliable revenue figures
- Manually curate a lookup table for top-revenue drugs

## Windows-Specific Issues Fixed

**Encoding error:** Python on Windows uses the system locale encoding (e.g., `cp950` on Traditional Chinese Windows) by default for file operations. Any Unicode character (e.g., `®`, `→`) in the data causes a `UnicodeEncodeError`.

**Fix applied:** All file read/write calls now explicitly specify `encoding="utf-8"`:
```python
# Correct
path.write_text(content, encoding="utf-8")
path.read_text(encoding="utf-8")

# Wrong (Windows default — breaks on non-ASCII characters)
path.write_text(content)
path.read_text()
```

## GitHub Actions Workflow

**File:** `.github/workflows/update-data.yml`
**Schedule:** `0 6 1,15 * *` — 6:00 AM UTC on the 1st and 15th of every month
**Manual trigger:** Go to GitHub repo → Actions → Update Antibody Data → Run workflow

**What the workflow does:**
1. Checks out the repository
2. Sets up Python 3.12
3. Installs `requests` (`pip install requests`)
4. Runs `python scripts/update_all.py`
5. Commits changed `data/*.json` files back to `main` branch (skips commit if no changes)

**Permissions required:** `contents: write` (set in the workflow YAML)

**If the workflow fails:** Go to Actions tab on GitHub, click the failed run, and read the logs. The most common causes are API rate limits or a changed API schema.

## Frontend (index.html / app.js / style.css)

**Tab system:** Driven by the `TABS` config array in `app.js`. Each entry specifies:
- `id`: matches HTML panel and element IDs
- `dataFile`: path to the JSON file to load
- `columns`: list of column definitions (key, label, sortable, filterable, badge type)

**To add a new column to an existing tab:**
1. Add an entry to the `columns` array for that tab in `TABS` (in `app.js`)
2. Add a corresponding filter `<select>` in `index.html` if `filter: true`

**To add a new tab:**
1. Add a new entry to `TABS` in `app.js`
2. Add a `<button class="tab-btn">` in the `<nav class="tabs">` in `index.html`
3. Add a `<section id="panel-catN">` with controls and table div in `index.html`
4. Create the corresponding `data/categoryN.json` and `scripts/fetch_categoryN.py`

**Search:** Full-text across all fields in the record using `String.prototype.includes`.
**Filters:** Exact-match dropdowns populated dynamically from unique values in the data.
**Sorting:** Click any column header; click again to reverse direction.

## Running Locally

# Install Python dependency (one-time)
pip install requests

# Run the full data pipeline
cd antibody-tracker
python scripts/update_all.py

# View the website locally
# Open index.html — NOTE: the browser will block fetch() for local files.

## Known Limitations & Future Work

| Area | Limitation | Potential fix |
|---|---|---|
| Category 1 indication | Not available from openFDA in structured form | Supplement from FDA label PDFs or a curated lookup |
| Category 1 patent status | Derived from marketing_status — not accurate | Use FDA Orange/Purple Book patent data |
| Revenue (all drugs) | 0/218 drugs have revenue data from public filings | Curated lookup table or paid API |
| Category 3 FDA CRLs | openFDA `submissions.action_type:CRL` query returns 0 results | Use FDA press releases or a different query |
| Category 3 date filter | Applied client-side (not at API level) | Check if ClinicalTrials.gov v2 adds a date filter parameter |
| Non-US companies | Many sponsors not in SEC EDGAR (Japanese, Korean, European companies) | Use company annual report URLs directly |
